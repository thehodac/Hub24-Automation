QC Automation — End-to-end process

Project: HUB24 / MPF QC framework · Author: Thong Ho · Date: 2026-07-20

The complete flow, from reading the project information to running cross-browser tests on real devices. Each step lists the tools, outputs, and commands to run.

1 Read project information  →  2 Generate Cucumber test cases  →  2b Push the test cases to the project's tool  →  3 Write automated tests  →  4 Run tests suite by suite (e2e / api / a11y / chromatic)  →  5 Push to GitHub + run CloudBees CI  →  6 Run BrowserStack (cross-browser / device)

1Read project information

The bot loads all known information about the project so it works from real documentation rather than guessing.

ApproachAsk which project this is, then read the entire projects-document/<project>/ folder (starting with project-info.md, then docs/). If the folder is empty → request the links (Confluence / doc URL, or web/app link) and/or ask that documentation files be added to the folder.
Toolread-project-docs skill
OutputWorking context: ticket tool (Jira/Asana/Azure/Basecamp), app URL (BASE_URL/API_URL), where the Func Spec lives, CI, conventions

2Generate Cucumber test cases

Read the Acceptance Criteria in the Functional Spec and convert them into Gherkin scenarios (Given/When/Then). The pipeline does NOT invent or duplicate AC — it only reads the AC already present in the spec.

ApproachCollect the spec input (Confluence / file), generate the scenarios; QC + BA review and approve before use
Toolgenerate-testcase / spec-to-testcase-pipeline skill (Claude Code)
Output.feature + .json in test-cases/ (source of truth)

2bPush the test cases to the project's tool

Once the test cases are generated, the bot reports that it is done and pushes them where the user wants — it never pushes on its own.

Approach 1. The bot reports the generation is done and asks: "Where do you want to push the test cases?" (Jira / Confluence / Asana — depending on the project's tool).
2. The bot asks the user to provide the link where the test cases should go (Jira issue link / Confluence page / Asana project). The bot does not guess it.
3. After the user enters the link, the bot pushes the test cases there (Excel .xlsx + JSON).
ToolJira / Confluence / Asana API (Claude Code)
OutputTest cases published at the link the user provided; the copy in test-cases/ remains the source of truth
NotePublishing is side-effectful — the bot confirms the destination + link with the user before pushing.

3Write automated tests

Turn the approved test cases into runnable Playwright specs, following the Page Object Model.

ApproachWrite specs in tests/e2e/ (1 test / scenario, keep the tags); put actions/locators into page objects under pages/; use shared fixtures
ToolPlaywright + TypeScript (Claude Code)
OutputSpecs in tests/ + page objects in pages/, committed
Checknpx tsc --noEmit + eslint

4Run tests suite by suite

Run each test type locally. On a failure, auto-create a bug ticket in the project's tool (with a clear reason, environment/device, test parameters); each run produces one pass/fail report.

Commands E2E / regression: npm run test:e2e
API: npm run test:api
Accessibility: npm run test:a11y
Visual (Chromatic): npm run test:visual:chromatic
ToolPlaywright · @axe-core/playwright · Chromatic · auto-ticket reporter
OutputResults + report in test-report/; bug ticket auto-created on failure ([E2E-Bug] / [API-Bug] / [Accessibility-Bug] / [Chromatic-Issue])

5Push to GitHub and run CloudBees CI

Commit and push the code; CI runs the whole suite automatically, auto-creates tickets on failure, and exports a report.

ApproachPush to the repo → the CloudBees pipeline runs e2e → api → a11y → chromatic, then auto-ticket + report. GitHub Actions mirrors this pipeline with a downloadable report artifact.
ToolGitHub + CloudBees (and GitHub Actions)
OutputCI results + auto-created tickets + report artifact

6Run BrowserStack (cross-browser / device)

Run the suite on real browsers and devices to confirm it works everywhere. Only run after the functional + visual steps pass, and this is the gate before promoting to PRE-PROD.

Commandsnpm run test:browserstack (or the CI gate after steps 4–5 pass)
ToolBrowserStack (real browsers & devices)
OutputCross-browser results; on failure → [BrowserStack-Bug] ticket with a session link; on pass → ready to promote to PRE-PROD

Notes

  • The copy in test-cases/ is always the source of truth even when test cases are also published to Confluence / Asana.
  • Where tickets are created depends on the project (Jira / Asana / Azure DevOps / Basecamp) — set it in .env; verify with node scripts/check-ticket-env.mjs.
  • Running for real against HUB24 (steps 3–6) is blocked until the HUB24 test environment is available (BASE_URL + credentials); for now the process is demonstrated on the SauceDemo sample project.
