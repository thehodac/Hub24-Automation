Automated workflow: Functional Spec → Test Case

Automation QC · Confluence (Project Nitro / PN1) · Related tickets: NIT-5129, NIT-5137 · Updated: 2026-07-07

Every day at 7am (VN time), a scheduled task reads the Functional Specifications on Confluence, generates test cases (Gherkin/Cucumber), publishes them to Confluence under "QC & Testing Tools", and saves a local copy into the test-cases/ folder in the automation repo.

Flow: Functional Specs (Confluence) → read User Story + Acceptance Criteria → generate Test Cases (positive / negative / edge / accessibility / mobile / analytics / gap) → Destination 1: Confluence "QC & Testing Tools" (the "Test Case | … (Excel+Json)" page + a Summary table per group) → Destination 2: local files test-cases/<group>/<feature>.json + .feature

2. Logic for each run

  1. List every feature spec (pages with a title of the form <Group> | <Feature>) under "Functional Specs". Skip empty container pages. (Archived pages automatically fall out of scope.)
  2. For each feature: read the User Story + Acceptance Criteria.
  3. Ensure the parent "<Group>: Test Cases" page exists (create it if missing).
  4. Decide: no test-case page yet → CREATE; spec modified more recently than the test-case page → UPDATE; otherwise → SKIP.
  5. Write the local .json + .feature copies (even in the skip case if the local file doesn't exist yet).
  6. Update the Summary table on the group's parent page (test-case count per feature + total, links as smart cards).

3. Test-case format (same as the App Header set)

  • Child Confluence page: a Test Case Table (Excel format) section (Scenario ID TC_<CODE>_NNN, Tags, Scenario, Description Given/When/Then) + a JSON Export section.
  • Tags: @ui @functional @accessibility @mobile @security @analytics (type), @positive @negative @edge (kind), @smoke (critical), @gap (spec gap/contradiction).

4. Local file structure

test-cases/
  README.md
  <group-slug>/
    <feature-slug>.json      # array of { scenario_id, tags, scenario, description }
    <feature-slug>.feature   # Cucumber/Gherkin, runnable

slug = lowercase, with spaces and the characters | ( ) / replaced by -. Example: "Main Page (Workspace)" → main-page-workspace.

5. Manual command

# Regenerate all .feature files from the .json files:
node scripts/json-to-feature.mjs

6. How to run

  • Run on demand in Claude Code: call the /spec-to-testcase-pipeline skill whenever a Functional Spec changes. Specs don't change daily, so there's no need for a scheduled run.
  • Requires the Atlassian (Confluence/Jira) connector to be connected — it comes from the claude.ai account, not from the repo.
  • No more scheduled task. The Cowork job daily-generate-test-specs (07:00 daily) was removed when moving to Claude Code — see plans/cowork-to-claude-code.html.

7. Notes

  • The single source for generation is the Functional Specs; test cases are derived — editing a spec and re-running updates the corresponding test cases.
  • An existing test-case page is only updated when the spec changes, to avoid unnecessary overwrites.
  • A new group in the Functional Specs → automatically creates the "<Group>: Test Cases" page and the corresponding local folder.
