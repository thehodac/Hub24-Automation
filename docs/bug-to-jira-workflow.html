Bug Report → Jira workflow (HUB24)

Automation: Playwright + custom reporter. Updated: 2026-07-09 · Jira project: NIT · Bug epic: NIT-5006

A custom reporter (reporters/bug-report.ts + reporters/jira.ts) automatically generates a bug report and syncs the Jira ticket after each test run.

1. When a test FAILS

  1. Print a summary to the console: Total · Passed · Failed.
  2. Export one .docx file per issue: bug-report/<type>/[FAIL][ISSUE-NN] <Func Spec> - <title>.docx. ISSUE-NN is stable and numbered separately per type.
  3. Save the screenshot → bug-image/<type>/ and the video → bug-video/<type>/.
  4. API tests (no browser): attach request/response via attachApiEvidence() → shown in the docx + rendered to PNG.
  5. Write the Change history (Vietnam time GMT+7) into the docx, adding a line for each run. The history is stored in the file only, not pushed to Jira.
  6. Create a Jira ticket with status New: if the parent is an Epic ⇒ create a Bug; if the parent is a Story ⇒ create a "Bug Fixing (in-sprint) Sub-task". Title [Bug] <Func Spec> - <title>, attach screenshot + video, automatically apply a severity label P1–P4 (see section 3b), deduplicate by label.

2. When re-run and the test PASSES

  1. Change the file prefix [FAIL] → [PASS] (docx, png, webm), keeping all content intact.
  2. Add a history line: which date it failed, which date it passed.
  3. Find the ticket by label and transition it to Done.

3b. Severity (labels P1–P4)

Each bug is automatically tagged with a Jira severity label (matching the QA/QC documentation standard), inferred from the test type / tag; it is also written into the .docx. This is an initial guess — QC re-confirms if needed.

4. Output folder structure

bug-report/  <type>/[FAIL|PASS][ISSUE-NN] <Func Spec> - <title>.docx
             <type>/.history/ISSUE-NN.json      (stores history + details)
bug-image/   <type>/[FAIL|PASS][ISSUE-NN] ... .png
bug-video/   <type>/[FAIL|PASS][ISSUE-NN] ... .webm

<type> = e2e | api | accessibility | chromatic | UI ...

5. Run commands

npm run test:e2e       # regression / e2e
npm run test:api       # API
npm run test:a11y      # accessibility
npm run test:visual    # chromatic / visual

# Run a specific file:
npx playwright test tests/e2e/<file>.spec.ts --project=e2e

6. Notes

  • "Func Spec" is taken from the outermost describe() — name the describe after the functional spec.
  • Tickets are deduplicated by label: the same test that is still open will not create a new ticket. A ticket already Done does not count as open → if it fails again, a new ticket is created.
  • The history evidence (fail/pass dates) stays in the file only, and is not added to the Jira ticket.
  • The demo files tests/**/_demo-fail*.spec.ts are only for testing the workflow — delete them when running for real.
  • Secrets stay in .env only (already gitignored); do not commit tokens.
