Chromatic → automatic ticket creation workflow (Visual regression)

Updated: 2026-07-16 · Run with a single command: npm run test:visual:chromatic

Goal: run the visual tests, create a new build on Chromatic, and when there is a visual change (test fail), automatically create a ticket (Jira & Asana) with the before + after images and the Chromatic build link (no video).

1. Running

npm run test:visual:chromatic

Required in .env:

CHROMATIC_PROJECT_TOKEN=chpt_...        # token of the CORRECT project on chromatic.com
JIRA_CREATE_TICKETS=true                # enable Jira ticket creation (+ JIRA_EMAIL/API_TOKEN/PROJECT_KEY/PARENT_KEY)
ASANA_CREATE_TICKETS=true               # (optional) also create an Asana task (+ ASANA_TOKEN/PROJECT_GID)

2. What the command does (3 internal steps)

StepAction
[1/3]Run the visual tests (tickets OFF) → produce local images (toHaveScreenshot) + Chromatic archives. No ticket yet because the build number is not known.
[2/3]Upload to Chromatic with chromatic --playwright --exit-zero-on-changes --force-rebuild → create a new build → save the link to .chromatic-build-url. Prints ✅ Chromatic build created -> <URL>.
[3/3]Run the visual tests again (tickets ON) → any test that differs from the baseline will fail → the reporter automatically creates a ticket with the before/after images + build link.

Why --force-rebuild is needed: by default Chromatic skips the build if the commit has not changed ("Skipping rebuild of an already passed/accepted build"). This flag forces a new build on every run.

3. When a visual test "fails"

toHaveScreenshot('...png') compares the current image with the local baseline (in tests/chromatic/*.spec.ts-snapshots/). If they differ → fail → create a ticket.

4. Managing the baseline

  • Create/update a clean baseline: npx playwright test --project=visual --update-snapshots (always passes, no ticket created).
  • Update just one test: add -g "TC_VIS_001".
  • The baseline must be the correct UI; if the baseline is "dirty" (captured by mistake while there was a change), regenerate it.

5. Ticket type (issue type) — IMPORTANT

A Chromatic ticket is not a Bug (a UI change is not necessarily an error). The ticket type depends on the type of the parent in .env (JIRA_PARENT_KEY):

Parent in .envChromatic (visual)Other types (e2e/api/a11y)
EpicStoryBug
StorySub-task (usually)Bug Fixing (in-sprint) Sub-task

Can be adjusted via env (optional): JIRA_CHROMATIC_ISSUE_TYPE (default Story), JIRA_CHROMATIC_SUBTASK_TYPE (default Sub-task), JIRA_CHROMATIC_TITLE_PREFIX (default [Visual]).

6. Ticket content

  • Title: prefix by test type — [E2E-Bug] / [API-Bug] / [Accessibility-Bug], except Chromatic which is [Chromatic-Issue] (no -Bug because it is not necessarily an error). E.g.: [Chromatic-Issue] SauceDemo — Visual regression - <test>.
  • Description: Reason / Failed step / Steps / Actual / Expected + the line Chromatic build: <URL>
  • Evidence: before (baseline) + after (current) images — no video.
  • Labels: automation, P4-minor, autobug-<hash> (deduplication).
  • Deduplication: an open ticket already exists for that test → skip, do not recreate.
  • Auto-close: when the test passes again → the ticket moves to Done (Asana: mark complete).

8. Distinguishing the commands (easily confused)

CommandCreate Chromatic build?Create ticket?
npm run test:visual❌ (no upload)✅ (before/after, no build link)
npm run chromatic✅ (upload only)❌
npm run test:visual:chromatic✅ (with --force-rebuild)✅ (before/after + build link)

9. Automation (no manual typing)

Add npm run test:visual:chromatic to CI (CloudBees / GitHub Actions) — a step is already added in .cloudbees/workflows/test.yaml. On each push → CI runs automatically → new build + ticket created automatically, without anyone typing a command. (Secrets are set in CI.)

10. Quick troubleshooting

  • No new build: missing --force-rebuild (already added) or looking at the wrong project/branch — open the correct ✅ Chromatic build created -> ... link.
  • No ticket: the test did not fail (baseline matches), or JIRA_CREATE_TICKETS≠true, or a duplicate ticket already exists.
  • Build went to the wrong project: the token in .env is not the project you are viewing (each project has its own token).
