BrowserStack → automatic ticket creation workflow (Cross-browser)

Updated: 2026-07-16 · Command: npm run test:browserstack (or run automatically in CloudBees)

Goal: run the test suite on real browsers in BrowserStack (multiple OS/browser). When a test fails → automatically create a Jira ticket with the prefix [BrowserStack-Bug], whose description contains a direct link to the session of that test so the developer knows exactly where the error is.

1. Running

npm run test:browserstack

Required in .env (running locally) or secrets (CloudBees):

BROWSERSTACK_USERNAME=...        # BrowserStack account
BROWSERSTACK_ACCESS_KEY=...      # access key
JIRA_CREATE_TICKETS=true         # enable ticket creation (+ JIRA_BASE_URL/EMAIL/API_TOKEN/PROJECT_KEY/PARENT_KEY)
BUG_CATEGORY_OVERRIDE=browserstack   # force the ticket to type "browserstack" -> [BrowserStack-Bug]

2. How it works

  1. The BrowserStack SDK (browserstack-node-sdk) runs the specified specs on real browsers in BrowserStack's cloud. Each test = a separate session.
  2. After each test, the captureBsSession(test) hook calls BrowserStack's getSessionDetails API to get the session's dashboard URL, and attaches it to the test (annotation browserstack-url).
  3. Test fails → the bug-report reporter reads that URL + gathers the evidence, and calls createBugTickets.
  4. Because BUG_CATEGORY_OVERRIDE=browserstack, every ticket from this run carries the browserstack type → prefix [BrowserStack-Bug], and the description has the line BrowserStack session: <url>.

Why getSessionDetails is needed: the local reporter does not know the session ID in the cloud. BrowserStack lets you retrieve session information via an "executor" (a no-op page.evaluate with a special JSON command), which returns browser_url = the dashboard link for the exact session.

3. Which specs run on BrowserStack

Defined by playwright.bs.config.ts (testMatch): the UI and accessibility specs (tests/UI/**, tests/accessibility/**). The real OS/browser come from browserstack.yml (the platforms section).

4. What the created ticket contains

  • Title: [BrowserStack-Bug] <Func Spec> - <test>
  • Description: Reason / Failed step / Steps / Actual / Expected + the line BrowserStack session: https://automate.browserstack.com/...
  • Evidence: image + video (like e2e/api).
  • Labels: automation, autobug-<hash> (deduplication), + severity label.
  • Deduplication + auto-close: an open ticket already exists for that test → skip; the test passes again → move to Done.

6. On CloudBees

The step "Cross-browser on BrowserStack (auto-create tickets)" in .cloudbees/workflows/test.yaml:

  • Uses the node:20-bookworm image (BrowserStack runs the browsers in the cloud, so no local browser install is needed).
  • Env: BROWSERSTACK_USERNAME/ACCESS_KEY, the JIRA_* variables, JIRA_CREATE_TICKETS=true, and BUG_CATEGORY_OVERRIDE=browserstack.
  • Has continue-on-error: true → a test failure does not block the other steps.
  • Runs npm run test:browserstack.

Secrets to declare on CloudBees: BROWSERSTACK_USERNAME, BROWSERSTACK_ACCESS_KEY, and the JIRA_* set (already present).

7. Quick troubleshooting

  • No ticket created: missing JIRA_CREATE_TICKETS=true/secrets, or the test did not fail, or a duplicate ticket already exists.
  • Ticket has no [BrowserStack-Bug] prefix: missing BUG_CATEGORY_OVERRIDE=browserstack in the step's env.
  • No "BrowserStack session" line: the captureBsSession hook was not called in the spec, or getSessionDetails did not return a URL (not running on BrowserStack).
  • Step is slow/hangs: normal because of the real cloud connection; if it hangs for a long time, check the BrowserStack account + whether browserstack.yml is valid.
