QC Report process — Test results & coverage

QC Backlog row 25 · Updated: 2026-07-09 · Metrics template: the "QC Metrics & Reporting" page (Confluence, row 11)

The scripts/qc-report.mjs script automatically aggregates the test results + bug counts by severity + test case counts, then produces the reports/qc-report.md report and evaluates the release criteria.

1. Data sources

MetricSourced from
Total / Passed / Failed / Pass ratePlaywright JSON report — merges all results/*.json (should cover all of e2e / api / accessibility / visual)
Open defects by P1–P4Jira — counts tickets labelled P1-blocker … P4-minor that are not Done (requires JIRA_* in .env)
Number of features / test casescounted in test-cases/**/*.json

2. How to run (each cycle)

npm run test:all     # e2e + api + accessibility + visual -> results/results.json
npm run qc:report    # -> reports/qc-report.md (aggregated)

test:all runs every type in a single pass so the report has full coverage. If you run each suite separately, each suite overwrites results/results.json → the report only has the last suite; to merge them, save each suite to its own file results/<name>.json (the script automatically sums every results/*.json file).

3. Report contents

  • Test results: Total · Passed · Failed · Flaky · Skipped · Pass rate % (target 100%).
  • Open defects by severity: P1 / P2 / P3 / P4 (from Jira).
  • Test cases: number of features + number of test cases.
  • Release exit criteria: Pass rate 100% (0 fail) · 0 P1&P2 open · Coverage ≥ 80%.

5. Notes

  • The "Open defects" section needs JIRA_BASE_URL / JIRA_EMAIL / JIRA_API_TOKEN / JIRA_PROJECT_KEY; if missing it is skipped (shows "—").
  • The 80% coverage needs to be reconciled against the acceptance-criteria baseline — currently left as "to confirm".
  • The report can be pushed to Confluence automatically (a future extension, not yet done).
