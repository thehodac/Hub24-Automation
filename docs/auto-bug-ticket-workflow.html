Automatic ticket creation workflow (Bug Report reporter)

Updated: 2026-07-17 · File: reporters/bug-report.ts · reporters/jira.ts · reporters/asana.ts

When any test fails, the reporter automatically creates a ticket in Jira (and Asana if enabled), attaches the evidence, and closes the ticket automatically when the test passes again. It runs both locally and in CI (CloudBees / GitHub Actions / Bamboo).

1. Flow

  1. Playwright runs the tests → the reporter gathers the results (onTestEnd / onEnd).
  2. Test fails → builds a "bug object" (reason, steps, environment, parameters, etc.).
  3. Deduplication: look for an open ticket for that test (by label). If one exists → skip; if not → create a new one.
  4. Create a ticket in Jira (and Asana if enabled) + attach image/video/evidence.
  5. Test passes again on a later run → the ticket is automatically moved to Done (Asana: mark complete).

2. Deduplication — what it relies on

Each test has a hidden label autobug-<hash>, where hash = md5(file-path + "::" + test-name). It is considered "already existing" when: the same project + the label is still present + status ≠ Done. → Changing the prefix/content has NO effect; to create a new one, set the old ticket to Done (or rename the test/file).

3. Ticket content (upgraded)

ItemContent
TitlePrefix by type + spec name + test name (section 4).
Reason (why it failed)A clear, easy-to-understand sentence (naming the element), e.g. "The 'add to cart …' element was expected to be visible, but it was not found." A11y → lists the number of WCAG violations. If the test has a custom message, that message is used.
Failed at stepThe step that failed (written in plain language).
Steps to reproduceThe steps in everyday language (e.g. "Enter username: standard_user", "Click the login button"), dropping the mechanical steps (Create context…). Passwords are masked automatically (***).
Actual / ExpectedActual result (clear sentence) vs expected.
Environment / DeviceDevice · OS + version · real browser (Chrome/Firefox/Safari + version). BrowserStack → the real device/OS/browser of the session.
Test parameters (user inputs)Inputs used for the test (e.g. user, module) — via attachTestParams(). Passwords are not included.
EvidenceImage + video. Chromatic → before/after + build link (no video). BrowserStack → session link. Accessibility → axe-results.json.
Test filePath of the spec that produced the bug.
Labelsautomation, autobug-<hash>, severity label (P1–P4).

5. Destinations: Jira + Asana

  • Jira (default): enable with JIRA_CREATE_TICKETS=true + credentials.
  • Asana (in parallel, optional): enable with ASANA_CREATE_TICKETS=true + token. Runs independently of Jira.

6. Configuration (environment variables)

# Jira
JIRA_CREATE_TICKETS=true
JIRA_BASE_URL=https://classltd.atlassian.net
JIRA_EMAIL=...            JIRA_API_TOKEN=...
JIRA_PROJECT_KEY=NIT      JIRA_PARENT_KEY=NIT-6655   # Epic or Story to attach the ticket to

# Asana (optional — runs in parallel with Jira)
ASANA_CREATE_TICKETS=true
ASANA_TOKEN=...
ASANA_PROJECT_GID=...     # Project ID (the number after /project/ in the URL) - REQUIRED
ASANA_SECTION_GID=...     # Section ID within the project - OPTIONAL (empty = default section)

# Report + CI
TEST_REPORT_DIR=...       # CI/BrowserStack sets a separate report folder

Getting the Asana GID: ASANA_PROJECT_GID = the number after /project/ in the project URL. Get ASANA_SECTION_GID with node scripts/asana-sections.mjs <PROJECT_GID> (prints the section name + GID). Leave it empty to place the task in the default section.

7. To include "Test parameters" in the ticket

The test attaches inputs itself via a helper, or uses the shared auto-fixture (every SauceDemo spec already has it via pages/saucedemo/fixtures.ts):

import { attachTestParams } from '../../utils/testMeta';
attachTestParams(testInfo, { username: 'standard_user', module: 'login' }); // mask secrets: password: '***'

For an even clearer error: use an assertion with a message → expect(header, 'Header not found').toBeVisible().
