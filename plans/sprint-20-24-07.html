4. Per-task approach (Didier's 5 phases each)

Row 32 — Asana: enable auto-create tasks

  1. Research: confirm the Asana reporter is wired (reporters/asana.ts); identify the required .env vars and where the token/GID come from.
  2. Diagram: add Asana as a bug destination on the auto-ticket sub-process diagram (docs/qc-subprocess-diagrams.html) — Jira and Asana in parallel.
  3. Test: fill .env (ASANA_CREATE_TICKETS=true + token + project GID), run node scripts/check-ticket-env.mjs, then run a deliberately failing test and confirm a task appears in the right Asana project/section.
  4. Document: record the result (screenshot of the created task, dedup on re-run, auto-complete on pass) in the workflow doc.
  5. Secure the flow: confirm ticket content (prefix, Environment/Device, params without password, clear error), then mark row 32 Done with evidence.

Done when: failing test → Asana task created in correct project · re-run → no duplicate · test passes → task auto-completed · ticket content verified.

Row 33 — Curate + load 1 robust MPF project into the knowledge base

  1. Research: with Kat, pick ONE robust past MPF project on Drive; agree what "good artefact" means (clean specs, well-formed test cases, feature files).
  2. Diagram: sketch how the agent uses the knowledge base when generating test cases (input → retrieve examples → generate) — a small flow in the docs.
  3. Curate + load: shortlist the quality artefacts, then load them into the agent's in-house / closed knowledge base (no client data leaves the environment).
  4. Test: run a generation against the loaded examples and check the output quality improves / follows the curated style.
  5. Document + secure: note which project + artefacts were used and the closed-env constraint; get Kat's confirmation. Mark Done.

Done when: 1 project curated, artefacts loaded in the closed KB, a test generation demonstrably uses them, Kat confirms.

Row 34 — Publish test cases to Asana

  1. Research: confirm the publish target (Asana project/link from the user) and the two files to attach (Excel .xlsx + JSON).
  2. Diagram: update the "Publishing generated test cases" step to show Asana as a destination alongside Confluence.
  3. Test: publish an approved sample set to Asana (ask destination + link first — this is side-effectful), verify the ticket + both files appear.
  4. Document: capture the created Asana item as evidence; note the repo copy stays source of truth.
  5. Secure the flow: confirm the confirmation-gate works (never auto-publish without the user's destination + link). Mark Done.

Done when: approved test cases published to the agreed Asana location with both files, after explicit user confirmation.

Row 35 — Demo & validation guideline (document + video)

  1. Research: list what to demo (e2e / api / a11y / chromatic + auto-tickets) and the audience.
  2. Diagram: reuse the end-to-end sub-process diagrams as the demo storyboard.
  3. Build: write a short guideline doc (steps to run the suite + what to look at) and record a short video walkthrough.
  4. Test: dry-run the guideline start-to-finish to make sure someone else could follow it.
  5. Secure the flow: run the validation session with the team, capture sign-off. Mark Done.

Done when: guideline doc + recording exist, dry-run passes, team validates and signs off.
