Plan — Make the repo runnable on the local machine

Author: Claude (Cowork) · Date: 2026-07-23 · Related: local dev setup, no dedicated ticket

1. Goal

Get Hub24-Automation to a state where it demonstrably runs end-to-end on this machine — dependencies installed, Playwright browsers installed, config sane, VS Code's Playwright Test extension can discover/run tests, and the SauceDemo suite passes green as proof. This does not unblock real HUB24 testing — that still needs BASE_URL + test credentials from Dev/Infra (backlog row 8), which is outside what a repo/config fix can solve.

2. Scope

  • In scope: verifying/installing Node deps + Playwright browsers, sanity-checking .env against what can actually run today, fixing one live safety issue found during the check (below), and proving the framework runs via the SauceDemo demo suite.
  • Out of scope: writing real HUB24 page objects/tests (blocked on the env); enabling Asana/Azure/Basecamp ticket destinations (separate backlog rows); changing CI (CloudBees/GitHub Actions/Bamboo); guessing or inventing a HUB24 BASE_URL/credentials.

3. What the read-only check already found

  • node_modules already populated (incl. playwright, chromatic, browserstack-node-sdk, tsc) — npm install has already been run on this machine.
  • Existing Chromatic snapshots under tests/chromatic/*-snapshots/ are named *-win32.png — confirms Playwright has already run successfully on this Windows machine before (browsers were installed and worked at least once).
  • .env exists and already has real values filled: CHROMATIC_PROJECT_TOKEN, BROWSERSTACK_USERNAME/ACCESS_KEY, and a real JIRA_API_TOKEN + email — node scripts/check-ticket-env.mjs would report Jira as "ready".
  • BASE_URL is still the placeholder https://playwright.dev — real HUB24 execution stays blocked, unchanged from before.
⚠ Safety issue found: .env currently has JIRA_CREATE_TICKETS=true with a real token, and the bug-report.ts reporter is wired into every test run (not just test:e2e). That means any failing test right now — including a demo/exploratory run — creates a real bug ticket in the live NIT Jira project. This is the exact risk the 2026-07-13 "Cowork → Claude Code" migration plan already flagged and decided to turn off for local runs; it looks like .env was never flipped back. Recommend setting it to false before any local run in this session, and only setting it back to true intentionally.

4. Step-by-step approach

  1. Fix the safety issue: edit .env → JIRA_CREATE_TICKETS=false. No other values touched (secrets stay as-is; file stays gitignored/uncommitted).
  2. Verify toolchain versions in a real terminal on this machine (VS Code's integrated terminal is fine): node -v (expect ≥22), git --version.
  3. Install Playwright browsers: npx playwright install — confirms/repairs the Chromium/Firefox/WebKit binaries this machine needs.
  4. Sanity-check the project compiles: npx tsc --noEmit and npx playwright test --list (should list tests with no config errors).
  5. Prove it runs end-to-end: run the safe demo suite — npm run test:saucedemo:demo (headed, slowed down, against the public saucedemo.com, not HUB24) — and confirm it passes.
  6. Confirm VS Code integration: open the Testing sidebar (Playwright Test extension) and confirm it lists the same tests / can run one.
  7. Re-run the ticket-env check (node scripts/check-ticket-env.mjs) to confirm it now reports "no destination enabled" (expected, since Jira was just turned off) — proving the safety fix took effect.
