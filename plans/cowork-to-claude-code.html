Plan — Migrate the project from Claude Cowork to Claude Code

Author: Thong Ho · Date: 2026-07-13 · Related: QC backlog (NITRO) · No dedicated ticket

1. Goal

The Hub24-Automation project is currently operated with Claude Cowork. From now on it moves to Claude Code. The goal of this task: bring the entire "AI automation" part (currently living inside Cowork, outside the repo) directly into the repo in a form Claude Code understands, and remove the pitfalls of Claude Code running directly on the local machine (unlike Cowork, which runs in the cloud).

Unchanged: the Playwright code, Page Object Model, the bug→Jira reporter, CloudBees CI, BrowserStack, Chromatic — all keep working with Claude Code.

2. Scope

In scope

  • (1) Skills: convert the 2 Cowork skill prompts in docs/ into real Claude Code skills in .claude/skills/, so the whole team can clone the repo and use them.
  • (3) Permissions: add .claude/settings.json with an allowlist of the safe, commonly used commands (npm test, npx playwright, read-only git…) — reducing how often Claude Code has to ask for permission.
  • (4) Fix the places that drifted from reality: README.md (still mentions the CI in .github/ that was removed), .env (currently has real Jira ticket creation enabled), CLAUDE.md (does not yet mention test-cases/ or the skills).
  • (2) Daily 07:00 job — REMOVE ENTIRELY. Decision 2026-07-13: no longer need a periodic run. This job lives in Cowork (the daily-generate-test-specs task) so it must be turned off in the Cowork app itself — the repo cannot delete it. In the repo: remove all descriptions of it, switch to "run on demand" via the /spec-to-testcase-pipeline skill.

    • Out of scope

    • The SauceDemo suite (just a sample demo — do not touch it).
    • Writing real page objects / specs for HUB24 — waiting for test environment access (backlog row 8, currently a BLOCKER).

    3. Step-by-step approach

    1. Create the 2 skills. Copy the contents of the 2 files docs/skill-prompt-*.md (which already have proper frontmatter name + description) into .claude/skills/<name>/SKILL.md. Keep the content as-is, only adjust 2 points to fit the Claude Code context:
      • spec-to-testcase-pipeline: the "Scheduling" section is currently written for Cowork ("runs only while the app is open") → change it to how it runs in Claude Code (type /spec-to-testcase-pipeline), noting that the periodic job will be decided later.
      • playwright-bug-to-jira: add a note that the reporter is already installed in the repo (reporters/bug-report.ts, reporters/jira.ts) — this skill is now mainly used to extend/modify, not to rebuild from scratch.
      The 2 original files in docs/ stay as documentation (do not delete).
    2. Create .claude/settings.json with permissions.allow for the read-only/safe commands commonly used in this repo. Do not allow destructive commands (git push, rm -rf, git reset --hard…).
    3. Edit .env: change JIRA_CREATE_TICKETS=true → false. Reason: Claude Code runs on your machine, and every test run that fails creates a real Jira ticket in the NIT project. Only set it to true when you intentionally want to create tickets.
    4. Edit README.md: remove/change the sentences still saying the CI is in .github/ (the "Structure" and "Visual regression" sections), point them to CloudBees (.cloudbees/workflows/test.yaml) — which is the CI actually in use.
    5. Update CLAUDE.md: add a short section about (a) the test-cases/ directory being the source of the "backlog" (18 features / 384 test cases generated from the Functional Spec), (b) the 2 newly created skills and when to use them, (c) the current state: no HUB24 environment yet → BASE_URL is still a placeholder, auth.setup.ts is still a TODO.
    6. Remove the 07:00 job from the docs. Remove all descriptions of the scheduled task in docs/spec-to-testcase-workflow.html, docs/skill-prompt-spec-to-testcase.md, test-cases/README.md and the spec-to-testcase-pipeline skill — replace with "run on demand". User's job: turn off the daily-generate-test-specs task in the Cowork app yourself.
    7. Re-check: run npx playwright test --list to make sure nothing is broken, and confirm the 2 skills appear in the Claude Code skill list.

    5. How to verify

    • npx playwright test --list → lists tests normally, no errors (proving the config is not broken).
    • In Claude Code, type / → see spec-to-testcase-pipeline and playwright-bug-to-jira in the skill list.
    • grep JIRA_CREATE_TICKETS .env → false.
    • grep -n "github" README.md → no more lines saying the CI is in .github/.
    • Try a command from the allowlist (e.g. npm test -- --list) → Claude Code no longer asks for permission.

    6. Risks / notes

    • The 07:00 job will stop running when you stop using Cowork. This is the only real loss — but it is not urgent: the spec does not change every day, and running it manually via the slash command is enough for now. Will discuss in step (2).
    • The Atlassian connector (Confluence/Jira) in Claude Code comes from the claude.ai account, not from the repo. The spec-to-testcase-pipeline skill only runs while this connector is connected. In background / cron runs, this connector may not be available — needs testing to be sure (related to step 2).
    • .env is not committed (already gitignored) → editing JIRA_CREATE_TICKETS only takes effect on your machine. If others on the team run the repo, they must be told to set it to false themselves. (.env.example already has false — correct.)
    • The allowlist in .claude/settings.json is committed → it affects the whole team. Only include safe commands; do not include git push, rm, or any command that writes to the system outside the repo.
