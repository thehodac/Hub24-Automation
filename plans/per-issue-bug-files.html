Plan — One file per issue, with [STATUS][ISSUE-NN] prefix

Author: MPF Developer · Date: 2026-07-03 · Changes the Bug Report reporter

1. Goal

Change the reporter so that instead of ONE combined .docx per run, each failed test produces its OWN set of files (report + image + video), named with a status + stable issue id + test name, and updated on re-runs.

2. Naming convention

Base name (same in all 3 folders): [STATUS][ISSUE-NN] <describe> - <test title>

  • STATUS = FAIL or PASS
  • ISSUE-NN = ISSUE-01, ISSUE-02, ... (stable per test across runs)
FolderExample file
bug-report/[FAIL][ISSUE-01] Demo bug report - intentionally fails....docx
bug-image/[FAIL][ISSUE-01] Demo bug report - intentionally fails....png
bug-video/[FAIL][ISSUE-01] Demo bug report - intentionally fails....webm

3. Behaviour per test, each run

  1. Fails, first time → assign next ISSUE-NN; create the 3 files as [FAIL][ISSUE-NN] <title>.
  2. Fails again (re-run) → find its existing files by title, keep the same ISSUE-NN, regenerate the .docx + copy the new screenshot/video, keep prefix [FAIL].
  3. Passes (was an issue before) → rename its existing 3 files' prefix to [PASS][ISSUE-NN]; keep the old evidence (no new capture on pass).
  4. Passes, never failed → do nothing.

No hidden state file — the issue id lives in the filename, so the folder itself is the source of truth. New number = highest existing ISSUE-NN + 1.

4. Files to change

FileChange
reporters/bug-report.tsReplace the single-combined-.docx output with per-issue files + the [STATUS][ISSUE-NN] naming + re-run update logic (also process passed tests).

5. Unchanged

  • Console summary (Total / Passed / Failed, steps to reproduce).
  • Jira ticket creation (still per failed test, when enabled).
  • screenshot / video capture on failure (config already set).

6. How to verify

  • Run the demo fail test → see [FAIL][ISSUE-01] ... in all 3 folders.
  • Fix the demo test (make it pass) and re-run → the same files become [PASS][ISSUE-01] ....
  • Add a second failing test → it gets [ISSUE-02].

7. Risks / notes

  • Old combined report files from before are left as-is; new runs use the new per-issue naming.
  • Matching is by the sanitised test title; renaming a describe/test starts a new issue id.
  • Nothing is auto-deleted.
