Plan — SauceDemo: write API + Accessibility + Chromatic (demo)

Author: Thong Ho · Date: 2026-07-15 · Follows on from the existing SauceDemo e2e suite

1. Goal

There is already an e2e suite for SauceDemo (36 tests). Now write the 3 remaining types — API, Accessibility, Chromatic (visual) also running against saucedemo.com, reusing the existing page objects, to demo all 4 types.

2. Scope

  • In scope: 3 new test files (accessibility, api, chromatic) targeting SauceDemo; reuse pages/saucedemo/ + utils/a11y.ts.
  • Out of scope: do not touch the existing e2e; do not change project config (use the api and visual projects, and a11y runs under chromium).

3. Each type — what it is based on

TypeFileBased on / how to do it
Accessibilitytests/accessibility/saucedemo-a11y.spec.ts Use the page objects to navigate to the pages (login, inventory, cart, checkout) → call checkA11y() to scan WCAG 2.1 A/AA.
APItests/api/saucedemo-api.spec.ts SauceDemo has no real REST API → use Playwright request to check at the HTTP level: GET pages/resources, assert status 200, content-type, body contains "Swag Labs".
Chromatic (visual)tests/chromatic/saucedemo-visual.spec.ts Import @chromatic-com/playwright; navigate to the pages (login, inventory) → toHaveScreenshot() to capture & compare against the baseline (before/after/diff).

4. Decisions needed (important for the demo)

  1. Accessibility: checkA11y() currently asserts 0 violations. SauceDemo may have real a11y issues → the test will FAIL.
    • Option A (recommended for the demo): leave it as-is — if it finds an issue, it is a good illustration of "a11y caught a bug → create a ticket".
    • Option B: loosen it (only assert serious/critical issues, or exclude a widget) to make the demo pass cleanly.
  2. API: since SauceDemo has no business API, the test is only an HTTP check (the page returns 200…). If you want a more "real" API demo, you could use a sample public API (reqres.in) — but by default follow the request: on saucedemo.com.
  3. Chromatic: the first run creates the baseline, later runs then compare the diff. Push to Chromatic using the existing test:visual:chromatic script.

6. How to run (projects already exist)

  • Accessibility: npm run test:a11y (filter SauceDemo: npx playwright test accessibility saucedemo)
  • API: npm run test:api
  • Chromatic/visual: npm run test:visual (push to Chromatic: npm run test:visual:chromatic)

7. How to verify

  • Run each command above → each type produces a result + an HTML report.
  • Force one case to fail (e.g. a11y finds an issue, or visual changes the baseline) → see the reporter note the bug in bug-report/<type> + (if enabled) push to Jira.

8. Risks / notes

  • a11y may genuinely fail on SauceDemo (see section 4.1) — choose option A or B.
  • api on SauceDemo is only an HTTP check (no business API).
  • The sandbox cannot run a browser → verify on your machine.
