Plan — API bug evidence: request/response + .png image

Author: MPF Developer · Date: 2026-07-06 · Related ticket: Epic NIT-5006

1. Goal

API tests have no browser, so there is no real screenshot/video. Goal: every API bug still has evidence — write the HTTP request/response into a .docx file, and render that information into a .png image saved in bug-image/api/ (then embed the image into the .docx). No video is recorded (there is nothing to record for API).

2. Scope

  • In scope: a helper for API tests to attach request/response; the reporter reads that evidence, adds it to the docx + renders it into a .png image; one demo API test with a fake call/HTTP so the image is visible.
  • Out of scope: video for API; changing how other test types run (e2e/chromatic/accessibility stay unchanged).

3. Step-by-step approach

  1. Evidence-collection helper — utils/apiEvidence.ts: the function attachApiEvidence(testInfo, { method, url, status, requestBody, responseBody, requestHeaders?, responseHeaders? }) formats it into readable text and calls testInfo.attach('api-evidence', { body, contentType: 'text/plain' }). Real API tests call this function after each request (or only when needed) to attach it.
  2. Reporter reads the evidence — in reporters/bug-report.ts: find the attachment named api-evidence on each failed case, read the text content (add an apiEvidence? field to FailedCase).
  3. Add to the .docx: a new "API request / response" section showing the request/response content.
  4. Render the .png image: use Playwright's built-in chromium (import('@playwright/test')), open it headless → setContent with the request/response content (as <pre>) → screenshot → save bug-image/api/[FAIL][ISSUE-01] ....png. Wrap in try/catch: if chromium is not available, skip the image, the docx still has the text.
  5. The image is saved with the correct name so the docx's existing image-embedding logic displays it automatically (no need to change the embedding part).
  6. Demo: edit tests/api/_demo-fail.api.spec.ts — create fake evidence (no network needed), attachApiEvidence(...) then let the test fail → run once to see the .docx + .png in the api folder.

4. Files to create / change

FileChange
utils/apiEvidence.tsCREATE — helper to attach request/response.
reporters/bug-report.tsEDIT — read the api-evidence attachment; add a section to the docx; render text→PNG using chromium (try/catch).
tests/api/_demo-fail.api.spec.tsEDIT — use the helper to attach fake evidence then fail (for the demo).

5. How to verify

  • Run: npx playwright test tests/api/_demo-fail.api.spec.ts --project=api
  • Expected: bug-report/api/[FAIL][ISSUE-01] ....docx has an "API request/response" section + the embedded image; bug-image/api/[FAIL][ISSUE-01] ....png appears.
  • No video for API (by design).

6. Risks / notes

  • Rendering the PNG requires chromium to be installed (npx playwright install chromium). If CI runs the api project without a browser installed → skip the image, the docx still has enough text (does not break).
  • Real API tests must call attachApiEvidence to have evidence; if not called, there is only the docx text as today.
  • The image is a "text image" (a rendered request/response), not a UI screenshot — because API has no UI.
